# Void Linux leg - multi-distro CI matrix (Phase A/B)

Worktree subagent leg, 2026-07-17. Brought up the Void Linux build+gate
environment (`ci/containers/Containerfile.void`), a clean in-container cmake
build, and the headless sceneprobe render, replicating the proven
Arch/Debian/Tumbleweed legs. Void is the minimalist source/rolling nerd leg
(xbps package manager), and there is NO ng docker reference for it, so every
package name below was resolved by prototyping with `xbps-query -Rs` against
the live repo. Scope: build + sceneprobe (B3). The e2e gate wiring (B2) is a
parallel agent's job and is not touched here. Branch
`multi-distro-ci-phase-a-void`.

## Test stage (ctest, recorded for A2 completeness)

`ci/build-and-gate.sh test` -> **80/82 pass** (98%). Only two failures, both
the known/expected classes and NEITHER a source defect:

- `schemesmodeltest` - the known non-hermetic test: it reads the ambient
  `XDG_DATA_DIRS` and picks up Void's real Breeze color schemes instead of
  its fixtures (the nix devShell's allow-listed XDG_DATA_DIRS hides this).
  Same failure Arch's leg recorded; needs a fixture-only XDG_DATA_DIRS to be
  portable. B2/gate-stage item.
- `qmllintgate` - the NixOS-tier qmllint RATCHET gate. Its baseline is
  calibrated to the pinned nix qmllint and it deliberately refuses a foreign
  qmllint, so it is excluded from the distro gate stage by design (a
  merge-gate, not a distro release-gate).

Notably this is BETTER than Arch (8 fails) and Tumbleweed (4 fails) at the
`test` stage: because this image exports `LATTE_QML_MODULE_PATH` and puts the
Qt6 QML host tools on PATH (Trap 3), the other QML gates
(qmlcompilegate/qmlinteraction/qmlcontracts) that those legs deferred already
run green here. Full e2e recipe characterization is still B2's job.

## Resolved runtime paths (the ENV block)

- lavapipe ICD: `/usr/share/vulkan/icd.d/lvp_icd.x86_64.json` (arch-suffixed
  name, like Tumbleweed - Arch ships plain `lvp_icd.json`)
- validation layers: `/usr/share/vulkan/explicit_layer.d`
  (VkLayer_khronos_validation.json present)
- framework QML tree: `/usr/lib/qt6/qml` (Arch-style non-multiarch). Note:
  `/usr/lib64` is a SYMLINK to `lib` on Void x86_64, so both
  `/usr/lib/qt6/qml` and `/usr/lib64/qt6/qml` resolve to the same tree; the
  canonical `/usr/lib` path is used. The build's `build/latte-qmldir.txt`
  (KDE_INSTALL_QMLDIR) came out under lib/qt6/qml, so the 00400f16c /
  18aac31b0 fixes carry the leg with no source change.
- Qt6 QML host tools: qmlcachegen/qmlimportscanner in `/usr/lib/qt6/libexec`
  (cmake finds by absolute path); qmllint/qmltestrunner in `/usr/lib/qt6/bin`
  (off default PATH - the image adds it to PATH for the ctest QML gates).

## Traps

1. **libsanitizer-devel split (the one portability gap).** Void SPLITS the
   gcc ASan/UBSan runtime + static preinit object (`libasan_preinit.o`,
   `libasan.so`, `libubsan.so`) into `libsanitizer-devel`, where
   Arch/Debian/Tumbleweed bundle them with the base gcc that `base-devel`
   pulls. The step-2.5 law builds every unit test with
   `-fsanitize=address,undefined` (`latte_add_unit_test`), so without this
   package the sanitized test targets fail to LINK ("cannot find -lasan /
   libasan_preinit.o") at [534/565]. Caught live in the Void container. This
   is a distro dep-split, NOT a source defect - fixed by adding
   `libsanitizer-devel` to the image (the runtime `libsanitizer` follows as
   its dependency). gcc and libsanitizer are both the 14.2.1 series, so they
   match.

2. **kwin_wayland cap_sys_nice=ep.** Identical trap to Arch (79a8008f0) and
   Tumbleweed: `/usr/bin/kwin_wayland` ships with the effective CAP_SYS_NICE
   bit (verified `getcap` showed `cap_sys_nice=ep` before strip) and podman's
   default cap set excludes it, so execve fails EPERM and the nested
   compositor never comes up. `setcap -r /usr/bin/kwin_wayland` (from
   `libcap-progs`) in the image strips it; `getcap` is empty after.

3. **Qt6 QML host tools off-PATH.** `qmllint` and `qmltestrunner` live in
   `/usr/lib/qt6/bin`, not on the default PATH (only qmlcachegen/
   qmlimportscanner, which cmake resolves by absolute path, are found without
   help). Same class as the Debian/Tumbleweed trap. The image adds
   `/usr/lib/qt6/bin` to PATH so the ctest QML gates find the tools by bare
   name; this does NOT shadow the build's Qt tools (cmake uses absolute
   IMPORTED_LOCATION for moc/rcc/uic). Does not affect the sceneprobe gate
   (it never invokes qmllint).

4. **base image ships no bash.** `void-glibc-full:latest` has `/usr/bin/sh`
   but no bash, and the entire gate stack is `#!/usr/bin/env bash`, so `bash`
   is a hard build dep here (not so on the other distros, whose base images
   carry it). Prototyping in the base image must use `sh -c`, not `bash -c`.

## F5 feed (future void-packages xbps template)

The build env this image proves is exactly what the Phase F5 xbps-src
`packaging/void/` template will need. Notable for that template's
`makedepends`: the KF6-`kf6-<name>-devel` naming, the split
`qt6-base-private-devel`, `libplasma-devel` (bare, not version-prefixed like
Tumbleweed's libplasma6), `kf6-kwayland-devel`, `layer-shell-qt-devel`, and
that `extra-cmake-modules` is bare (no kf6- prefix). `libsanitizer-devel` is
a checkdepend (only the sanitized tests need it), not a runtime dep.

## Reproduce

From the repo root (or the worktree checkout):

    podman build -t latte-void -f ci/containers/Containerfile.void .
    SRC=$PWD
    podman run --rm -v "$SRC":/src:ro -v latte-void-build:/build -w /src \
        latte-void bash /src/ci/build-and-gate.sh build      # 565/565
    podman run --rm -v "$SRC":/src:ro -v latte-void-build:/build -w /src \
        latte-void bash /src/ci/build-and-gate.sh test        # ctest (QML-gate class deferred to B2)
    podman run --rm -v "$SRC":/src:ro -v latte-void-build:/build -w /src \
        latte-void bash -c 'BUILD=/build bash /src/scripts/sceneprobe-gate.sh'  # 13/13 PASS bit-exact
